User Permissions

Understanding File Permissions and Ownership

  • After you have worked with Linux for a while, you are almost sure to get a Permission denied message.
    • If not, try this command (to be executed as non-root user): cd /root/
  • Permissions associated with files and directories in Linux were designed to keep users from accessing other users' private files and to protect important system files.
  • File permissions are a fundamental security feature in Linux. They help:
    • Protect user privacy by preventing unauthorized access to files.
    • Maintain system integrity by restricting modification of critical system files.
    • Enable collaboration by allowing controlled access to shared files.
  • The nine bits assigned to each file for permissions define the access that you and others have to your file.
  • Permission bits for a regular file appear as -rwxrwxrwx.
  • Those bits are used to define who can read, write, or execute the file.

Note

For a regular file, a dash appears in front of the nine-bit permissions indicator.
Instead of a dash, you might see a d (for a directory), l (for a symbolic link), b (for a block device), c (for a character device), s (for a socket), or p (for a named pipe).

  • Of the nine-bit permissions, the first three bits apply to the owner's permission, the next three apply to the group assigned to the file, and the last three apply to all others.

  • The r stands for read, the w stands for write, and the x stands for execute permissions.

  • If a dash appears instead of the letter, it means that permission is turned off for that associated read, write, or execute bit.

  • Because files and directories are different types of elements, read, write, and execute permissions on files and directories mean different things.

Permission File Directory
Read View what's in the file. See what files and subdirectories it contains.
Write Change the file's content, rename it, or delete it. Add files or subdirectories to the directory. Remove files or directories from the directory.
Execute Run the file as a program. Change to the directory as the current directory, search through the directory, or execute a program from the directory. Access file metadata (file size, time stamps, and so on) of files in that directory.

Owner

  • The owner of a file is typically the user who created it.
  • They have the most control over the file's permissions.
  • Think of it like owning a car; you can decide who gets to drive it, who can ride in it, and who can't touch it at all.

Group

  • Files in Linux can be assigned to a group.

  • Think of groups in Linux as exclusive clubs or teams.

    • Each file and directory on your system is assigned to a specific group.
    • Just like a club has members, a group has users as its members.
    • Now, here is the key: members of a group can have different permissions on a file than those who are not in the group.

  • Groups provide a granular level of control over file access.

    • Imagine you are working on a project with a team.
    • You want your team members to be able to edit and modify project files, but you don't want anyone else on the system to mess with them.
    • This is where groups come in handy!
    • By assigning the project files to a specific group and adding your team members to that group, you can grant them the necessary permissions (like read and write) while restricting access for others.

Types of Groups

  • Linux has different types of groups:
    • Primary Group:
      • Each user has a primary group, usually with the same name as the username.
      • When a user creates a file, it is automatically assigned to their primary group.
    • Secondary Groups:
      • A user can belong to multiple secondary groups.
      • This allows for more flexible permission management in various scenarios.
    • System Groups:
      • Linux has built-in system groups for specific purposes, like wheel (for administrative access) or audio (for managing audio devices).

Managing Groups

  • You can use commands like groups to see a user's groups, groupadd to create new groups, groupdel to delete groups, and usermod to add or remove users from groups.

Others

  • Everyone else who is not the owner or a member of the group falls into the "others" category.

  • These are like the general public when it comes to your file.

  • You can set permissions to allow them some access (like read-only), or you can completely restrict their access.

  • In the context of file permissions in Linux, "others" refers to any user on the system who is not the owner of the file and does not belong to the group associated with the file.

  • These are essentially the "outsiders" when it comes to accessing the file.

  • Even though you might primarily be concerned about the permissions for yourself (the owner) and your group, it's still important to consider the permissions for "others."

    • Here is why:
      • Public Access:
        • If you want to make a file accessible to everyone on the system, you need to grant appropriate permissions to the "others" category.
        • For example, if you have a script that you want anyone to be able to execute, you would grant execute permission to "others"
      • Security:
        • By default, files are often created with some level of permission for "others."
        • If you don't want anyone outside of the owner and group to have any access to the file, you need to explicitly remove the permissions for "others."
        • This is important for protecting sensitive or confidential information.
      • Shared Resources:
        • In some cases, you might have files or directories that are intended to be shared with a broader audience, but not necessarily everyone on the system.
        • By carefully setting permissions for "others," you can control the level of access they have, such as read-only or execute-only.

Viewing Permissions

  • As noted earlier, you can see the permission for any file or directory by typing the ls -ld command.
    • -l: use a long listing format.
    • -d: list directories themselves, not their contents.
  • The named file or directory appears as those shown in this example:
$ ls -ld ch3 test
-rw-rw-r-- 1 joe sales 4983 Jan 18 22:13 ch3
drwxr-xr-x 2 joe sales 1024 Jan 24 13:47 test
  • The first line shows that the ch3 file has read and write permission for the owner and the group.
  • All other users have read permission, which means that they can view the file but cannot change its contents or remove it.
  • The second line shows the test directory (indicated by the letter d before the permission bits).
  • The owner has read, write, and execute permissions while the group and other users have only read and execute permissions.
  • As a result, the owner can add, change, or delete files in that directory, and everyone else can only read the contents, change to that directory, and list the contents of the directory. (If you had not used the -d options to ls, you would have listed files in the test directory instead of permissions of that directory.)
  • Additionally:
    • joe is the owner.
    • sales is the group associated with the file/directory.
    • 4983 and 1024 is the size of the file/directory in bytes.
    • Jan 18 22:13 and Jan 24 13:47 is the last modification to the file/directory.

Changing Permissions with chmod (Numbers)**

  • If you own a file, you can use the chmod command to change the permission on it.
  • In one method of doing this, each permission (read, write, and execute) is assigned a number — r=4, w=2, and x=1 — and you use each set's total number to establish the permission.
    • For example, to make permissions wide open for yourself as owner, you would set the first number to 7 (4+2+1), and then you would give the group and others read-only permission by setting both the second and third numbers to 4 (4+0+0), so that the final number is 744.
  • Any combination of permissions can result from 0 (no permission) through 7 (full permission).

Examples

  • The following chmod command results in this permission: rwxrwxrwx
$ chmod 777 filename.ext
  • The following chmod command results in this permission: rwxr-xr-x
$ chmod 755 filename.ext
  • The following chmod command results in this permission: rw-r--r--
$ chmod 644 filename.ext

The following chmod command results in this permission: ---------

$ chmod 000 filename.ext

More on chmod

  • The chmod command also can be used recursively.
  • For example, suppose that you wanted to give an entire directory structure 755 permissions (rwxr-xr-x), starting at the $HOME/myapps directory.
  • To do that, you could use the -R option, as follows:
$ chmod -R 755 $HOME/myapps
  • All files and directories below, and including, the myapps directory in your home directory will have 755 permissions set.

Exercises

  1. Permission Investigation:

    • Use the ls -l command to examine the permissions of various files and directories on your system.
    • Try to understand the permission strings and what they imply about access rights.

  2. File Lockdown:

    • Create a file named secret.txt and make it so that only you (the owner) can read and write to it.
    • No one else, not even those in your group, should have any access.

  3. Group Collaboration: Create a directory named project and set the permissions so that:

    • You have full control (read, write, execute).
    • Members of the group developers have read and write access. (You should create that group -- i.e. groupadd)
    • Everyone else has no access at all.

  4. Executable Script:

    • Write a simple bash script, save it as myscript.sh, and then make it executable for everyone.

  5. Directory Exploration:

    • Set the permissions on a directory so that anyone can list its contents and access files within it, but only you (the owner) can create new files or delete existing ones within that directory.

  6. Recursive Change:

    • You have a directory named archive containing many subdirectories and files.
    • Change the permissions of the entire directory structure (including all subdirectories and files) so that everyone has read-only access.
