UNIX filesystem

  • The UNIX filesystem, which forms the basis for Linux filesystems, is a crucial component that distinguishes UNIX and Linux from other operating systems.
  • The Linux filesystem, derived from the UNIX filesystem.
  • It adheres to the principle that "everything is a file," meaning that directories, devices, and even processes are represented as files within the system.
  • It is designed with a few key principles in mind:
    • This unified approach simplifies interactions, allowing the use of standard commands and tools for various system components.
    • The Linux filesystem is organized hierarchically, resembling an inverted tree with the root directory (/) at the top.
    • This structure enables logical organization and easy navigation of files and directories.
  • Key directories within the Linux filesystem include:
    • /bin: Contains essential user commands.
    • /etc: Stores system-wide configuration files.
    • /home: Houses the home directories of individual users.
    • /lib: Contains shared libraries required by system programs.
    • /var: Stores variable data, such as log files and spool directories.
    • /usr: Contains user programs, libraries, and data not essential for system boot.

Benefits

  • Logical Organization: The hierarchical structure allows for intuitive organization of files and directories, making it easy to find and manage data.
  • Flexibility and Extensibility: The "everything is a file" approach provides a consistent interface for interacting with various system components, simplifying system administration and enabling the development of new tools and applications.
  • Efficiency and Power: The use of small, specialized commands and the ability to combine them using pipes and redirection allows for efficient and powerful automation of tasks.
  • Security: The filesystem includes permissions and ownership controls that allow for fine-grained access control to files and directories, ensuring data security and privacy.

Key Directories

  • There are several key directories within the UNIX filesystem, each serving a specific purpose:
    • /bin: This directory houses essential user commands that are frequently used for basic operations and system interactions. Commands like ls (list directory contents), cat (concatenate and display files), cp (copy files), mv (move or rename files), and rm (remove files) reside here, making them readily accessible to all users.
    • /boot: This directory contains critical files required for booting the Linux system. The Linux kernel itself, along with the initial RAM disk (initrd) and bootloader configuration files (such as those for GRUB), are stored here.
      • These files are crucial for the system's startup process and should be handled with care.
    • /dev: This directory contains special files that represent devices connected to the system.
      • These include physical devices like hard drives (/dev/sda), terminals (/dev/tty*), and CD-ROMs (/dev/cd*), as well as virtual or pseudo-devices like random number generators (/dev/random).
      • Applications can interact with these devices through their corresponding files in the /dev directory.
    • /etc: This directory stores system-wide configuration files.
      • These files control the behaviour of various system services and applications, and they are typically plain text files that can be edited with a text editor (with appropriate permissions).
      • Examples include /etc/passwd (user account information), /etc/fstab (filesystem mount table), and /etc/apache2/apache2.conf (Apache web server configuration).
    • /home: This directory contains the home directories of individual users on the system.
      • Each user has their own subdirectory within /home, where they can store their personal files, configuration settings, and other data.
      • The root user's home directory is located at /root for security reasons.
    • /lib: This directory houses shared libraries that are used by various applications and system programs.
      • These libraries provide common functionalities and routines that can be reused by multiple programs, promoting code efficiency and modularity.
    • /media: This directory serves as a standard location for automatically mounting removable media, such as USB drives and external hard disks.
      • When a removable device is connected, it is typically mounted under a subdirectory within /media, making its contents easily accessible to users.
    • /mnt: While largely superseded by /media for automounting, /mnt is still commonly used for manually mounting filesystems, such as those on additional hard drives or network shares.
      • It provides a flexible location for temporarily accessing storage devices that are not part of the system's default configuration.
    • /opt: This directory is intended for storing optional or add-on application software.
      • It provides a standardized location for third-party applications that are not part of the base Linux distribution, making it easier to manage and maintain these additional software packages.
    • /proc: This is a special virtual filesystem that provides information about the system's running processes and kernel parameters.
      • It is not a physical storage location, but rather an interface to the kernel, allowing users and applications to access real-time system data.
    • /root: This is the home directory of the root user, the superuser account with full administrative privileges.
      • It is located outside /home for security reasons, as it contains sensitive configuration files and system data.
    • /sbin: This directory contains system administration commands and daemons, which are typically used by the root user or other privileged users to manage and configure the system.
    • /snap: This directory is the home for filesystems used by the snapd package management system.
      • Snaps are self-contained application packages that include all their dependencies, making them easy to install and manage.
    • /sys: This is another virtual filesystem that provides an interface to the kernel, specifically for accessing and managing system hardware and device information.
    • /tmp: This directory is used for storing temporary files created by applications and system processes.
      • These files are typically deleted automatically after a certain period or upon system reboot.
    • /usr: This directory contains user programs, libraries, and data that are not essential for the system's boot process.
      • It is often mounted as a separate filesystem and can be shared across multiple systems using network file sharing protocols like NFS.
    • /var: This directory stores variable data that is expected to change frequently during system operation.
      • This includes log files (/var/log), spool directories (such as /var/spool/mail for email), and other data generated by system services and applications.

Comparison to Windows Filesystems

  • The UNIX filesystem differs from Windows filesystems in a few key ways:
    • No Drive Letters: Unlike Windows, which uses drive letters (C:, D:, etc.) to represent different storage devices, UNIX mounts all storage devices into a single hierarchical structure, providing a unified view of the filesystem.
    • Forward Slashes: UNIX uses forward slashes (/) as path separators, while Windows uses backslashes ().
    • Permissions and Ownership: UNIX has built-in permissions and ownership controls for files and directories, allowing for fine-grained access control to files and directories, while Windows relies on file and folder attributes for access control.
      • Each file and directory has associated permissions for the owner, group, and others, specifying who can read, write, and execute them.

inode

  • Inodes (short for "index nodes") are the unsung heroes of the Linux filesystem.
  • They are like hidden metadata files that store essential information about each file and directory on your system.
  • Think of them as the control panels behind the scenes, keeping track of everything related to a file except its name and actual data.

What is Inside an Inode?

  • Each inode typically contains the following information:

    • File Type: Whether it is a regular file, directory, symbolic link, etc.
    • Permissions: Who can read, write, and execute the file (owner, group, others).
    • Owner and Group: The user and group associated with the file.
    • Size: The size of the file in bytes.
    • Timestamps: When the file was last accessed, modified, and its inode changed.
    • Data Block Pointers: These pointers tell the filesystem where the actual data for the file is stored on the disk.

  • Inodes play a crucial role in the Linux filesystem:

    • Unique Identity: Each file has a unique inode number, allowing the system to distinguish between files even if they have the same name.
    • Efficient Access: Inodes provide a quick way for the system to locate and access file information without having to scan the entire disk.
    • Data Organization: Inodes help organize the file data by maintaining pointers to the data blocks.
    • File System Operations: Inodes are involved in various file system operations, such as creating, deleting, renaming, and linking files.

Inodes and Links

  • Inodes are also closely related to the concept of hard links.
  • When you create a hard link, you are essentially creating another filename that points to the same inode as the original file.
  • This means both filenames share the same file information and data blocks.

Working with Inodes

  • To access the inode metadata of a file, you can use the stat command.
  • The stat command is a powerful tool for retrieving detailed information about a file, including its inode metadata.
  • Think of it as a supercharged ls -l command that dives deeper into the file's properties.
  • The basic syntax is simple:
stat filename
  • Additionally, ls -i can show the Inode number of a file.
$ ls -i myfile.txt
2376895 myfile.txt
  • You can also use the find command to search for files by their inode number.
  • For example, the following command will find all files with inode number 2376895:
$ find . -inum 2376895
  • This command will search the current directory and all subdirectories for files with inode number 2376895.

Example

  • Let us say you have a file named myfile.txt.
  • To view its inode metadata, you would run:
stat myfile.txt
  • The output of stat will look something like this:
File: 'myfile.txt'
Size: 1234 Blocks: 8 IO Block: 4096 regular file
Device: 801h/2049d Inode: 1234567 Links: 1
Access: (0644/-rw-r--r--) Uid: (1000/johndoe) Gid: (1000/johndoe)
Access: 2024-03-25 10:00:00.000000000 -0400
Modify: 2024-03-26 11:00:00.000000000 -0400
Change: 2024-03-27 12:00:00.000000000 -0400
 Birth: -
  • The output provides a wealth of information, including:
    • File: The name of the file.
    • Size: The size of the file in bytes.
    • Blocks: The number of blocks allocated to the file on the disk.
    • IO Block: The size of the input/output blocks used for the file.
    • Device: The device on which the file is stored.
    • Inode: The inode number of the file.
    • Links: The number of hard links to the file.
    • Access: The access permissions of the file (owner, group, others).
    • Uid: The user ID and username of the file's owner.
    • Gid: The group ID and group name of the file's group owner.
    • Access: The last time the file was accessed.
    • Modify: The last time the file's content was modified.
    • Change: The last time the file's attributes (metadata) were changed.
